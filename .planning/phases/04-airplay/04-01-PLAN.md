**Part A — Add updateTxtRecord to ServiceAdvertiser (Pitfall 1 fix):**

In `ServiceAdvertiser.h`, add a new pure virtual method:
```cpp
// Update a single TXT record key for an already-advertised service.
// Used by AirPlayHandler to replace the placeholder pk value with the real Ed25519 public key.
// serviceType: e.g., "_airplay._tcp"
// key: TXT record key to update (e.g., "pk")
// value: new value for that key
virtual bool updateTxtRecord(const std::string& serviceType,
                             const std::string& key,
                             const std::string& value) = 0;
```

In `AvahiAdvertiser.cpp`, implement `updateTxtRecord`:
- Find the stored entry group for the given serviceType
- The AvahiAdvertiser needs to store the per-service TXT records and entry groups. If the current implementation uses a single entry group, it must be refactored to track per-service data so individual TXT records can be updated.
- To update: call `avahi_entry_group_reset()` on the group, then re-add the service with the modified TXT records, then call `avahi_entry_group_commit()`.
- If the AvahiAdvertiser currently stores all services in a single entry group, the simplest approach is to store the full service registration data (type, name, port, txt vector) and re-register ALL services when any TXT record changes. This is acceptable since TXT updates are rare (once at startup when pk is known).

In `DiscoveryManager.h`, add a public method:
```cpp
// Update a TXT record value for an already-running service (Pitfall 1).
// Called by AirPlayHandler after generating/loading the Ed25519 keypair.

                     const std::string& key,
                     const std::string& value);
```

In `DiscoveryManager.cpp`, implement it by delegating to `m_advertiser->updateTxtRecord(...)`.
